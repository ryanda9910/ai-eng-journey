Week 1 selesai. trade-note, CLI yang mengubah kalimat bebas soal trade jadi JSON schema-valid lewat forced tool schema, dinyatakan PASS oleh grader independen. Bukan saya yang menilai: grader menjalankan kodenya sendiri dari nol dengan 5 input pilihan dia (campuran Bahasa Indonesia dan Inggris, termasuk edge case field yang tidak disebut).

Hasilnya: 5 dari 5 input keluar JSON valid dengan exit code 0, log token dan biaya muncul di stderr. Suite test retry lulus 3 dari 3, termasuk mock 429 yang recover di attempt kedua dan kasus schema gagal terus-menerus yang exit 2 tanpa bocor apa pun ke stdout.

![Run live trade-note di backend Groq plus hasil test retry]({{ '/shots/2026-07-14-w1-pass.png' | relative_url }})

Satu bonus yang tidak direncanakan: di run live kedua, Groq beneran menolak tool schema sekali, dan loop re-ask recover di attempt 2. Jadi jalur re-ask terbukti di kondisi nyata, bukan cuma di mock.

## Angka

- 5/5 input grader: JSON valid, exit 0
- 3/3 test retry/re-ask lulus tanpa API key (semua mocked)
- Biaya per call di backend Groq (llama-3.3-70b): sekitar $0.0003, tercatat per call di stderr
- Satu kejadian schema-rejection live, recovered lewat re-ask

## Yang gagal dan pelajarannya

Rencana awal pakai Anthropic API. Semua key lokal habis kredit, jadi saya tambah backend Groq (wire format OpenAI-compatible, forced function tool_choice, raw httpx tanpa SDK) di belakang interface yang sama. Grader menerima ini karena semangat aturannya, direct API tanpa framework, tetap terpenuhi. Jalur Anthropic tetap default kalau key terisi.

Pelajaran paling penting justru soal desain schema: spec mewajibkan side itu long atau short, tidak boleh null. Akibatnya input "cut loss di posisi ADA" memaksa model menebak side=long dengan confidence 1. Schema yang melarang "tidak tahu" akan dijawab dengan tebakan yang terlihat yakin. Kalau sebuah field bisa unknown, buat nullable.

Week 2 berikutnya: golden set minimal 20 kasus plus script eval yang menilai akurasi structured output.
